<task type="auto">
  <name>Task 1: Implement sync CLI command</name>
  <files>scanner/cli.ts, scanner/sync.ts</files>
  <action>
1. Create scanner/sync.ts — extract the core sync logic from src/app/api/sync/route.ts into a reusable module. The functions cloneRepo, pullRepo, and isCleanWorkingTree should be refactored to:
   - Accept a console logger callback instead of SSE writer
   - Return results instead of streaming them
   - Keep the same git clone/pull/ff-only logic

2. In cli.ts, add sync handler for `pm sync [clone|pull|both]`:
   - Default action is "both" if no argument provided
   - Validate action argument
   - Read LOCAL_PROJECTS_DIR env var (required, exit with error if missing)
   - Read manifest.json (required, exit with error if missing)
   - Build work list same as API route does
   - Execute clone/pull operations with console output showing progress
   - Print summary table at end: repo name, action, result

3. Update the API route (src/app/api/sync/route.ts) to import from scanner/sync.ts instead of having inline logic. This avoids code duplication between CLI and UI.

Note: Use execFileSync for git clone (not execSync) to prevent command injection — the API route already does this correctly for clone. For pull, use execSync with cwd option (safe since it only runs `git pull --ff-only` with no user input in the command string).
  </action>
  <verify>
Run `npm run pm sync` and verify it attempts to clone missing repos and pull existing ones. Run `npm run pm sync clone` and verify clone-only behavior. Check that the API route still works after refactor.
  </verify>
  <done>
- scanner/sync.ts contains reusable sync logic
- `npm run pm sync [clone|pull|both]` works
- API route imports from scanner/sync.ts (no duplication)
- Console output shows progress per repo
- Summary printed at end
  </done>
</task>

<task type="auto">
  <name>Task 2: Implement overrides CLI commands</name>
  <files>scanner/cli.ts, scanner/overrides.ts</files>
  <action>
1. Create scanner/overrides.ts — shared overrides logic:
   - loadOverrides(): reads data/overrides.json
   - saveOverrides(overrides): writes data/overrides.json
   - updateOverride(projectName, updates): merges updates into project's overrides
   These functions are already partially implemented in both scanner/index.ts (loadOverrides) and src/app/api/overrides/route.ts. Consolidate into one place.

2. In cli.ts, add overrides handler with two sub-subcommands:
   - `pm overrides list` — Print all overrides as a formatted table (project name, status, tags, notes preview)
   - `pm overrides set <project> [--tag <tag>] [--status <status>] [--notes <text>]`:
     - Parse flags from remaining args
     - Validate status is one of: active, recent, stale, abandoned, paused (or empty to clear)
     - --tag can be repeated to set multiple tags
     - Update overrides.json
     - Print confirmation of what was set

3. Update scanner/index.ts and src/app/api/overrides/route.ts to import from scanner/overrides.ts instead of having their own loadOverrides implementations.
  </action>
  <verify>
Run `npm run pm overrides list` and verify table output. Run `npm run pm overrides set myproject --tag test --status paused --notes "testing"` and verify overrides.json is updated. Verify API route still works.
  </verify>
  <done>
- scanner/overrides.ts with shared load/save/update logic
- `pm overrides list` shows formatted table
- `pm overrides set` updates overrides.json correctly
- Both CLI and API route use shared overrides module
- Status validation works
  </done>
